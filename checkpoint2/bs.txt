==============================================================
PHASE 2 — RICHARD: SYMBOL TABLES + SCOPES + UNDEFINED/REDEFINED
==============================================================

A) Symbol Table Core Data Structures
- Implement a scope stack + per-scope hash table:
  - Scope = HashMap<String, Symbol>
  - Stack of scopes supports nested blocks/functions
- Provide core operations:
  - enterScope(), exitScope()
  - insert(symbol) into current scope
  - lookup(name) searching outward through scopes
  - lookupCurrent(name) in current scope only

B) Symbol Model (must support type-checking later)
- Implement Symbol data structure with fields like:
  - name
  - kind: var / array / function / parameter
  - base type: int / bool / void
  - array size (if array)
  - function signature:
    - return type
    - parameter list types (and param count)
  - declaration position (line/col) for good error messages

C) Scope Rules (fully nested)
- Implement correct scope entry/exit for:
  - global scope
  - function scope (and parameters)
  - compound blocks { ... } inside functions (nested)
- Ensure nested scopes behave correctly:
  - inner declarations can shadow outer names
  - but redeclaration in SAME scope is an error

D) Semantic Errors: Undefined and Redefined
- Redefined identifier:
  - same name declared twice in SAME scope => error
- Undefined identifier:
  - using a variable/array/function name not found in any visible scope => error
- Keep going after errors (do not crash; report and continue)

Deliverable for Phase 2:
- Symbol tables build correctly for valid programs
- Correct nesting + lookup behavior
- Undefined/redefined errors reported consistently
- Symbols contain enough info for Phase 3 type checking

Integration point:
- Tameem depends on lookup(name) returning correct type/signature info
- Muhammad depends on an entry method to run symbol collection + optional -s printing

==============================================================
PHASE 3 — TAMEEM: TYPE CHECKING + SEMANTIC ANALYZER VISITOR + RECOVERY
==============================================================

A) Type Model + Helper Functions
- Implement Type representation:
  - INT, BOOL, VOID, ERROR  (ERROR helps recovery)
- Helper checks:
  - expectInt(type, context)
  - expectBool(type, context)
  - compatible(lhs, rhs) for assignments
  - operator rules for unary/binary operations
- Ensure type errors don’t crash the run (return ERROR and continue).

B) SemanticAnalyzer Post-order Visitor (Type Checking)
- Traverse AST in post-order; return a type for expressions.
- Implement rubric-required checks:
  1) Array index must be int
  2) Assignment LHS and RHS must be compatible
  3) Operands of binary/unary operations must be compatible
  4) Function calls/returns:
     - call arg count matches param count
     - each arg type matches expected param type
     - return statements match function return type
  5) If/while test conditions are int/bool (per assignment notes)

C) Error Reporting Format (match Phase 1 contract)
- All semantic errors go to stderr:
  - line:col: Semantic error: <brief message>
- Use AST node positions (or stored positions) to include line/col.

D) Semantic Error Recovery (continue after errors)
- Do NOT stop at first semantic error.
- Strategy:
  - If mismatch or undefined symbol, report error and return ERROR type.
  - Parent nodes continue; reduce “cascading” duplicate errors when possible.

Deliverable for Phase 3:
- SemanticAnalyzer.java completes type checking for all required rules
- Reports multiple semantic errors in one run
- Integrates cleanly with Phase 2 symbol table + Phase 1 CLI/-s output
