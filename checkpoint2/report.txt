## Richard — PHASE 2 (Symbol Tables + Scopes + Undefined/Redefined)

### What was done
- Implemented a symbol table with nested scopes for the C- AST.
- Wrote a semantic analyzer visitor that walks the AST and fills the symbol table.
- Added checks and error messages for undefined and redefined identifiers and wired `-s` to print scopes.

### Design decisions
- Used a stack of hash maps (one map per scope) to keep scope handling simple.
- Represented symbols with a small `Symbol` class plus enums for kind (var/array/function/param) and base type.
- Printed each scope’s symbols right before leaving the scope, with indentation for nesting level.

### Lessons learned
- Thinking in terms of a scope stack makes it easier to reason about nested blocks and functions.
- Built-in functions like `input` and `output` need entries in the symbol table so user code type-checks cleanly.
- Small, focused tests (1.cm–5.cm) are very helpful for validating symbol table behaviour.

### Assumptions / limitations + improvement ideas
- Phase 2 only handles undefined/redefined identifiers; full type rules are left for Phase 3.
- Built-in functions are hardcoded in the analyzer instead of being read from a separate library description.
- Symbol table printing is minimal; later we could print more detail like declaration locations or parameter names.

### Anything interesting you want to say?
- It was satisfying to see the printed symbol tables match the structure of real C- programs.
- Debugging both locally and on the school server showed how much tooling and environment affect compiler work.

## Tameem — PHASE 3 (Type Checking + Semantic Analyzer + Recovery)

### What was done
- Implemented full type checking in SemanticAnalyzer.java using a post-order visitor pattern.
- Added type tracking via `lastType` and `lastIsArray` fields that propagate types up the AST after each visit.
- Implemented all five marking-scheme type checks: array index must be int, assignment type compatibility, operator operand types, function call argument count/type matching, and return type validation.
- Added checks for if/while conditions (must be bool or int), void variable declarations, array-vs-scalar mismatches in function arguments, and using a non-array as an array.
- Fixed ShowTreeVisitor to accept a PrintWriter parameter so the `-a` flag actually writes the AST tree to the output file instead of stdout.
- Added 7 new error recovery rules to CM.cup (if/else condition, while condition, return expression, parenthesized expression, function call arguments, malformed function at global level), bringing the total from 5 to 12, since we lost marks for this previously.
- Removed unused `Letter` macro from CM.flex and added an `ERROR` token production in CM.cup to eliminate build warnings.

### Design decisions
- Used `lastType` and `lastIsArray` instance fields to pass type information back up from child nodes, since the visitor interface returns void. This avoids changing the AbsynVisitor interface.
- Tracked the current function's return type with `currentFunctionReturnType` so return statements can be validated against the enclosing function.
- Used an ERROR base type for recovery: when a type error is found, ERROR is returned so parent nodes can skip redundant error messages and avoid cascading false positives.
- Kept operator type rules close to the C- spec: arithmetic operators require int operands and produce int, relational operators require int and produce bool, equality accepts same-type operands and produces bool, logical operators accept bool or int and produce bool.

### Lessons learned
- Propagating an ERROR type after the first error in a sub-expression prevents cascading duplicate messages.
- Function prototypes (forward declarations) required special handling: replace the prototype symbol rather than reporting a redefinition error.
- Array parameters declared as `int a[]` have size 0, so `isArray` is the reliable way to distinguish them from scalars, not the size field.

### Assumptions / limitations + improvement ideas
- The type checker does not verify that every path through a non-void function has a return statement; it only checks that the return statements that exist have the correct type.
- Logical operators (`&&`, `||`, `~`) accept both bool and int operands per the assignment spec; a stricter version could require bool only.
- Void parameters (e.g., `int f(void)`) are handled at the grammar level and don't reach the type checker as actual parameter declarations.
- A future improvement could add warnings for unused variables or unreachable code after return statements.

### Anything interesting you want to say?
- The trickiest part was function call validation: checking arg count, each arg's type, and whether each arg is an array or scalar all at once, while still producing helpful error messages for each mismatch.
- Seeing the type checker catch 10 different errors in 5.cm in a single pass and continue cleanly was a good sign that the error recovery approach works.
